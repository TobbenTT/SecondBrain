[pie opcional: referencias a issues]

Ejemplo Completo

feat: agregar autenticacion con Google OAuth

Se implementa el flujo OAuth 2.0 con Google como proveedor.
Esto permite a los usuarios iniciar sesion sin crear cuenta manual.

Closes #42

Reglas: Explicar el que y el porque, nunca el "como". Max 72 caracteres. Minusculas despues del tipo. Un commit = un cambio logico.

5 Protocolo de Integracion (Pull Requests)

Ningun desarrollador puede subir codigo directamente a main. Todo cambio debe pasar por un Pull Request con revision obligatoria.

Checklist de Aprobacion

✅CompilacionEl codigo compila sin errores
🔒SeguridadNo hay credenciales ni llaves API expuestas
📝DocumentacionSe actualizo si fue necesario
🧪TestsLas pruebas pasan y se agregaron nuevas
🎨EstiloSigue las convenciones del proyecto
📏AlcanceSolo cambios relacionados a la tarea

Plantilla de Pull Request

## Descripcion
<!-- Que cambio se realizo y por que? -->

## Issue Relacionado
<!-- Closes #numero -->

## Tipo de Cambio
- [ ] Nueva funcionalidad (feature)
- [ ] Correccion de error (bugfix)
- [ ] Refactorizacion
- [ ] Documentacion
- [ ] Mejora de rendimiento

## Como se probo?
<!-- Describir los tests realizados -->

## Checklist
- [ ] Mi codigo compila sin errores
- [ ] No expongo credenciales ni secretos
- [ ] Actualice la documentacion si fue necesario
- [ ] Agregue tests para los nuevos cambios
- [ ] Todos los tests existentes pasan

7 Flujo Diario del Desarrollador

Paso Accion Comando
1️⃣Actualizar localgit checkout main && git pull origin main
2️⃣Crear ramagit checkout -b feature/nombre-tarea
3️⃣Registrar avancesgit add . && git commit -m "feat: descripcion"
4️⃣Publicar ramagit push origin feature/nombre-tarea
5️⃣Crear PRDesde la interfaz de GitHub

Ejemplo Practico Completo

# 1. Actualizar main
git checkout main
git pull origin main

# 2. Crear rama para la nueva tarea
git checkout -b feature/formulario-contacto

# 3. Desarrollar y hacer commits atomicos
git add src/components/ContactForm.tsx
git commit -m "feat: agregar estructura del formulario de contacto"

git add src/services/emailService.ts
git commit -m "feat: implementar servicio de envio de email"

git add src/tests/contactForm.test.ts
git commit -m "test: agregar pruebas unitarias del formulario"

# 4. Publicar la rama
git push origin feature/formulario-contacto

# 5. Ir a GitHub -> Crear Pull Request -> Asignar revisores

8 Mejores Practicas

✅ Hacer (Do)

  • ●
    Commits pequenos y atomicos
    Facilitan el review y el rollback
  • ●
    Mensajes descriptivos
    Historial claro y trazable
  • ●
    PRs enfocados en una tarea
    Reviews mas rapidos y efectivos
  • ●
    Actualizar rama con main frecuentemente
    Evitar conflictos grandes al merge
  • ●
    Eliminar ramas despues del merge
    Repositorio limpio y organizado
  • ●
    Usar .gitignore adecuadamente
    No subir archivos innecesarios

❌ No Hacer (Don't)

  • ●
    Push directo a main
    Codigo sin revisar en produccion
  • ●
    Mensajes como "fix", "update"
    Historial incomprensible
  • ●
    PRs con cientos de archivos
    Imposible de revisar efectivamente
  • ●
    Subir credenciales o llaves API
    Vulnerabilidad de seguridad critica
  • ●
    Trabajar dias sin hacer commits
    Perdida de trabajo y commits enormes
  • ●
    Ignorar comentarios del code review
    Deuda tecnica acumulada

10 Tarjeta de Referencia Rapida

🔀 Ramas

git checkout -b feature/nombre    # Crear feature
git checkout -b bugfix/nombre     # Crear bugfix
git branch -d feature/nombre      # Eliminar local
git push origin --delete feature/nombre  # Eliminar remota

📝 Commits

git add .                                  # Agregar cambios
git commit -m "feat: descripcion"          # Commit feature
git commit -m "fix: descripcion"           # Commit bugfix
git commit --amend -m "nuevo msg"          # Corregir ultimo

🔄 Sincronizacion

git pull origin main              # Actualizar main
git merge main                    # Integrar en tu rama
git push origin feature/nombre    # Publicar tu rama

🏷️ Tags y Releases

git tag -a v1.0.0 -m "Release v1.0.0"  # Crear tag
git push origin v1.0.0                  # Publicar tag
git tag -l                              # Listar tags
